# DevOpsLab1/Lab2

Цей репозиторій містить вихідний код веб-застосунку (Node.js + Prisma) та скрипти для його повністю автоматизованого розгортання на віртуальній машині під управлінням Linux з використанням Nginx, MariaDB та Systemd Socket Activation.

**Варіант:** V2 = 1; V3 = 3 ; V5 = 4;

**Опис завдання:** Розгортання трирівневої архітектури веб-застосунку (API, Reverse-Proxy, База даних) на віртуальній машині. Cтворення користувачів `student`, `teacher`, `operator`, `mywebapp`, автоматичне виконання міграцій бази даних перед запуском та оптимізація споживання ресурсів за допомогою механізму Systemd Socket Activation.

---

## 2. Документація по розробленому веб-застосунку

### Призначення застосунку
**Simple Inventory API** — це RESTful веб-застосунок для базового управління інвентарем. Він дозволяє зберігати інформацію про товари (назва, кількість) у базі даних, створювати нові записи та переглядати існуючі за допомогою HTTP-запитів.

### Налаштування середовища (Development)

Для локальної розробки або тестування поза віртуальною машиною необхідно:

1. Встановити Node.js (v18+) та npm.
2. Встановити MariaDB/MySQL та створити локальну базу даних.
3. Клонувати репозиторій та виконати команду:
   npm install
4. Створити файл .env у корені проекту та вказати рядок підключення до БД:
DATABASE_URL="mysql://USER:PASSWORD@localhost:3306/inventory_db"
5.Виконати міграцію для створення таблиць:
npx prisma migrate dev
6. Запустити командою : node index.js --port 8000

### 3. Документація по API-ендпоінтах

**Застосунок надає наступні маршрути:**

Бізнес-логіка (Публічні маршрути):
GET / — Головна сторінка (HTML-інтерфейс із посиланнями).
GET /items — Отримання списку всіх товарів (підтримує формати HTML та JSON).
GET /items/:id — Отримання детальної інформації про конкретний товар за його ID.
POST /items — Створення нового товару. Очікує JSON-тіло: {"name": "string", "quantity": number}.

Службові маршрути (Доступні лише локально):
GET /health/alive — Liveness probe. Повертає 200 OK (перевірка, що процес Node.js живий).
GET /health/ready — Readiness probe. Повертає 200 OK, якщо є успішне з'єднання з базою даних MariaDB, інакше 500 Database connection failed.

### 4.Документація по розгортанню
Образ віртуальної машини
ОС: Ubuntu 24.04 LTS (або 22.04 LTS) Server Edition.

Джерело: Офіційний образ від Canonical (завантажується у вигляді .iso для VirtualBox/VMware або стандартний образ для хмарних провайдерів).

Вимоги до ресурсів ВМ
CPU: 1 Core (мінімум).

RAM: 1 GB (рекомендовано 2 GB для комфортної роботи npm/Prisma).

Disk Space: 10 GB (стандартна розбивка). 

**Як завантажити та запустити на чистій віртуальній машині**:
1.  Виконати в консолі `sudo apt install git`
2. Створити відповідну папку та клонувати туди репозиторій
3. Перейти у папку репозиторію та у файлі setup.sh та в кінці файлу змінити miracle (використовувася для тестування роботи скрипту) на назву вашого базового користувача (user/ubuntu etc.)
4. У консолі запустити скрипт наступними командами :

`sudo chown +x setup.sh`

`sudo ./setup.sh`

Скрипт самостійно встановить пакети, створить користувачів, налаштує БД, розгорне код у /var/www/mywebapp та налаштує Nginx і Systemd. Також після виконання скрипту стандартний користувач (вказаний у пункті 3) буде заблокований

**Після завершення роботи setup.sh коректність роботи перевіряється наступним чином:**

Перевірка ізоляції користувачів:

Перевірка наявності користувача з його групою: `id operator; id student; id teacher`.

Перевірка створення файлу варіанту: `sudo cat /home/student/gradebook `(має вивести 14839938).

### Одразу після розгортання перевірте статус сервісу:
`systemctl status mywebapp.service`

Очікується статус: inactive (dead).

Перевірте статус сокета:

`systemctl status mywebapp.socket`

Очікується статус: active (listening).

## 5. Запуск проєкту за допомогою Docker Compose (Лабораторна робота №2)

В рамках другої лабораторної роботи архітектуру проєкту було контейнеризовано для забезпечення максимальної портативності. 

Замість systemd, застосунок працює у вигляді трьох взаємопов'язаних Docker-контейнерів:
1. db (`inventory_db`): База даних MariaDB.
2. web (`inventory_app`): Node.js застосунок який збирається через Dockerfile.
3. nginx (`inventory_nginx`): Reverse-proxy сервер.

### Особливості архітектури
* **Ізольована мережа:** Усі сервіси піднімаються у власній кастомній мережі `inventory_network` (driver: bridge). Вони спілкуються між собою за внутрішніми DNS-іменами, не виставляючи зайві порти назовні.
* **Автоматична міграція (Race Condition fix):** Для гарантованого створення структури БД використовується скрипт `entrypoint.sh`. Він очікує повного запуску MariaDB (через `netcat`), виконує `npx prisma migrate deploy`, і лише після цього передає управління Node.js.

Також, щоб уникнути конфлікту із системним Nginx (з Лабораторної №1), зовнішній порт для Docker-версії змінено на `8080`.

### Інструкція із запуску

1. Переконайтеся, що на машині встановлено `docker` та `docker compose`.
2. Перейдіть у корінь репозиторію.
3. Виконайте команду для збірки образів та запуску контейнерів у фоновому режимі:
   ```bash
   docker compose up -d --build

## 2. Інструменти тестування
Для реалізації тестового середовища було використано:
* **Jest** — фреймворк для написання, запуску юніт-тестів та генерації звітів покриття (`coverage`).
* **Supertest** — інструмент для тестування HTTP-маршрутів без необхідності реального підняття сервера на мережевому порту машини CI.
